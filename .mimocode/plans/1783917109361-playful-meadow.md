## Audit Summary

### Technology Stack

| Layer | Technology |
|-------|-----------|
| Backend framework | FastAPI (async) |
| ORM | SQLAlchemy 2.0 (async, `AsyncSession`) |
| Validation | Pydantic v2 + pydantic-settings |
| Auth | python-jose (HS256 JWT) + passlib/bcrypt |
| Database | SQLite (dev) / PostgreSQL 16 (prod) |
| Migrations | Alembic (configured but no versions exist) |
| Frontend | React 18 + TypeScript 5.6 + Vite 5 |
| Styling | Tailwind CSS 3.4 (dark mode via `class`) |
| Routing | react-router-dom v7 |
| Icons | lucide-react |
| Charts | Pure SVG (no chart library) |
| Containerization | Docker + docker-compose (3 services: db, api, frontend) |
| Background jobs | None configured (placeholder `events/` module is empty) |

### Entry Points

- **Backend**: `backend/app/main.py` — FastAPI app with lifespan (auto-create tables + seed categories on startup). Mounts 7 routers under `/api/v1`.
- **Frontend**: `frontend/src/main.tsx` → `App.tsx` — 9 routes, 8 protected behind `ProtectedRoute`.
- **Docker**: `docker-compose.yml` — 3 services (Postgres, API on :8000, Frontend via nginx on :3000).
- **Alembic**: `backend/alembic/` — configured for async Postgres but `versions/` is empty (no migrations).

### Data Flow (Traced: Create Priority)

```
Frontend (Priorities.tsx)
  → prioritiesApi.create()
  → api.post('/priorities', data)              [api.ts]
  → Vite proxy /api/* → http://127.0.0.1:8001 [vite.config.ts]

FastAPI (main.py → api_router → priorities.py)
  → Pydantic PriorityCreate validates body
  → Depends(get_db) injects AsyncSession
  → Depends(get_current_user) validates JWT     [deps.py → security.py]
  → add_priority(db, user.id, payload)          [priority_gate.py]
    → COUNT(*) active priorities on track
    → If ≥ 1: raise PriorityCapError → 409
    → Else: INSERT + COMMIT → PriorityRead JSON

Frontend receives response → refreshes priority list
```

| domains | `/domains` | list, create, rescore | Yes |
| audits | (no prefix) | reports, audit-log | Yes |
| learning | `/learning` | tracks, items, routines, activity | Yes |

### 10 SQLAlchemy Models

`User`, `Category`, `TimeBlock`, `Priority`, `Quarter`, `Deadline`, `ProtectedWindow`, `TargetDomain`, `AuditReport`, `AuditLogEntry` + 4 learning models (`LearningTrack`, `CurriculumItem`, `PracticeRoutine`, `PracticeLog`).

### Key Business Rules Implemented

1. **Priority Gate** — 1 active TECHNICAL + 1 active LANGUAGE cap (service layer)
2. **Block Guard** — flags protected window + cutoff violations on time blocks (never rejects)
3. **Cadence State Machine** — REST → REVIEW → SPRINT → CLOSED quarterly cycle
4. **Deadline Enforcement** — force-ship at due_date (service exists but is never triggered)
5. **Niche Filter** — score domains against core assets (scoring logic is broken — see issues)
6. **Learning Streaks** — consecutive-day tracking with rest-day tolerance

### Test Coverage

32 tests across `backend/tests/`:
- Auth (5), Priorities (4), Time Blocks (3), Cadence (3), Learning (6)
- Unit: Priority Gate (4), Learning Streak (7)

### Architecture Strengths (to preserve)

1. Clean separation: models / schemas / services / routers are well-isolated
2. `UUIDStrMixin` elegantly solves UUID serialization across dict and ORM input
3. Priority Gate correctly enforced at service layer with proper 409 responses
4. Block Guard correctly flags without rejecting, matching spec
5. Learning module well-structured: idempotent seeding, ownership checks, pure streak function
6. All DB queries use SQLAlchemy parameterized builders (no SQL injection risk)
7. Auth pattern consistent: every protected route uses `Depends(get_current_user)`
8. Dark mode flash prevention correctly implemented via inline script + class toggle
